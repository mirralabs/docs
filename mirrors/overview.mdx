Mirra's catalog focus is **application infrastructure** — the vendors your product depends on: payments, email, SMS, auth, commerce, databases, inference.

Each mirror is a real implementation of the vendor's public API running inside Mirra. Every mirror ships with a fidelity test suite, a public drift-detection SLA, and explicit documentation of what it does and doesn't cover.

## What's explicitly **not** on the roadmap

- GitHub (issues, PRs, actions)
- Slack (messaging, channels)
- Discord, Jira, Linear
- Google Workspace / Microsoft 365

These are productivity-tool territory — a different category. If you need mirrors for those, other tools cover that ground.

## Mirror properties — what every mirror has

Every mirror in the catalog ships with:

- **Per-session SQLite** — full state isolation
- **Vendor-correct webhook signing** — the exact HMAC scheme, header name, and signed-content layout the vendor uses
- **State machine fidelity** — for entities with lifecycles (subscriptions, messages, emails, auth sessions)
- **Idempotency** — where the vendor enforces it, the mirror does too
- **Named built-in fixtures** — curated starting states per mirror (e.g., `resend:transactional-busy`, `stripe:subscription-lifecycle`)
- **Public fidelity suite** — captured real-API fixtures replayed on every commit
- **Weekly drift scans** — comparison against the live vendor
- **Status page entry** — at [status.mirra.run](https://status.mirra.run) with last-scan timestamp and open drift issues
